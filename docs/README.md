# Документация проекта «Европротокол ДТП»

## 1. Назначение проекта

«Европротокол ДТП» — многостраничное веб-приложение для электронной фиксации дорожно-транспортных происшествий.

Проект хранит сведения о ДТП, участниках, автомобилях, повреждениях, фотографиях, свидетельских показаниях и факте отправки карточки в ГИБДД.

## 2. Технологический стек

- Python 3.12+
- Django 5.2 LTS
- PostgreSQL
- Django ORM
- Django Templates
- HTML, CSS, Bootstrap 5
- Pillow
- python-dotenv

## 4. Архитектура приложения

Проект разделен на два основных Django-приложения:

- `users` — отвечает за пользователей, профили водителей, автомобили профиля и проверку автомобильных данных;
- `accidents` — отвечает за ДТП, участников, автомобили в ДТП, повреждения, фотографии, свидетельские показания, отправку и аналитику.

Конфигурация проекта находится в `config`. Общий шаблон расположен в `templates/base.html`, а стили интерфейса — в `static/css/style.css`.

## 5. Основные модели данных

### `DriverProfile`

Профиль водителя. Хранит ФИО, телефон и номер водительского удостоверения.

### `DriverVehicle`

Автомобиль, сохраненный в профиле водителя. При добавлении автомобиля в ДТП данные копируются из этой модели.

### `Accident`

Основная карточка ДТП. Хранит название, дату, место, коды доступа, создателя и дату отправки.

### `AccidentDriver`

Связь водителя с ДТП. Хранит роль участника, комментарий, статус готовности и время подтверждения.

### `Vehicle`

Автомобиль внутри конкретной карточки ДТП. Связан с ДТП и профилем водителя.

### `Damage`

Повреждение автомобиля. Привязано к конкретному автомобилю в рамках ДТП.

### `AccidentPhoto`

Фотография ДТП, загруженная участником. Может быть привязана к конкретному автомобилю.

### `WitnessStatement`

Свидетельское показание. Хранит ФИО, телефон, текст показаний и необязательную фотографию.

### `SubmissionLog`

Журнал отправки ДТП в ГИБДД.

## 6. Связи между сущностями

- `DriverProfile` связан с `User` отношением один-к-одному.
- `DriverVehicle` связан с `DriverProfile` отношением многие-к-одному.
- `Accident` связан с создателем через `created_by`.
- `AccidentDriver` связывает `Accident` и `DriverProfile`.
- `Vehicle` связан с `Accident` и владельцем `DriverProfile`.
- `Damage` связан с `Vehicle`.
- `AccidentPhoto` связан с `Accident`, загрузившим пользователем и опционально с `Vehicle`.
- `WitnessStatement` связан с `Accident`.
- `SubmissionLog` связан с `Accident` и пользователем, выполнившим отправку.

## 7. Бизнес-правила

- ДТП имеет отдельный код доступа для свидетелей.
- ДТП имеет отдельный код подключения второго водителя.
- Один водитель не может быть добавлен в одно ДТП дважды.
- В карточке ДТП должно быть два участника.
- Каждый участник подтверждает готовность отдельно.
- Отправка доступна только после подтверждения обоими участниками.
- После отправки карточка блокируется для дальнейшего редактирования через пользовательский интерфейс.
- Автомобиль в ДТП выбирается только из автомобилей, сохраненных в профиле текущего водителя.
- Добавление или изменение данных участника сбрасывает его готовность.

## 9. Валидация данных

В проекте проверяются:

- государственный номер автомобиля;
- VIN;
- номер полиса ОСАГО;
- код подключения второго водителя;
- код доступа свидетеля;
- принадлежность выбранного автомобиля текущему пользователю.

Проверка автомобильных данных вынесена в `users/vehicle_validation.py`. Формы используют эти функции при очистке полей.

## 12. ООП и SOLID

В проекте используется объектно-ориентированный подход:

- модели Django представлены классами;
- формы наследуются от `forms.Form` и `forms.ModelForm`;
- общие настройки форм вынесены в `BootstrapFormMixin`;
- методы `clean()` и `save()` переопределяются для валидации и создания связанных объектов.

Принципы SOLID соблюдаются частично:

- модели отвечают за структуру данных;
- формы отвечают за ввод и валидацию;
- представления отвечают за обработку HTTP-запросов;
- вспомогательные функции вынесены отдельно;
- зависимости от Django ORM остаются прямыми, что нормально для проекта такого масштаба.

## 16. Краткое резюме

Проект реализует хранение и обработку данных электронного европротокола. Архитектура разделена на приложения `accidents` и `users`, данные хранятся в PostgreSQL, пользовательский интерфейс построен на Django Templates, а основные сценарии покрыты тестами.
